Abstract

This report describes all of the updated aspects and diagrams regarding the final implemented solution for our WADE project, Beverage Manager. It contains all of the updated sections from the middle term report as well as new sections describing the project progress, technical aspects regarding the implemented APIs, considerations regarding the RDF-based knowledge model and use of external-data sources and examples of sparql queries.

Keywords

  • Ontology
  • Semantic Web
  • Linked Data Principles

Introduction

The project consists in multiple services, each with their own functionality, thus creating the SOA architecture. Those services can be clustered into 4 categories: front end, back end, data related and database.

The application exposes all of its features to the user with the help of HTML5 pages loaded with dynamic and static data. The pages will be offered through the BeM Frontend service. The data will come through the Gateway Service, a service to which the Frontend will make all the requests. The gateway will further ask for data from other services that have their own way to query and format data from the database.

Our application will use a single knowledge base both for user and beverages (meta-)data storage. This knowledge base will interact with our Search & Filter, Beverage Profile, User Profile and Recommendations services. The process of data collection is described in section 8, whereas the steps for creating the knowledge base are discussed in section 10. A description of the ontological model is given in section 9.

Final Users

The target for the application was to have a place where people can search with ease through beverages and find offers, recipes, reviews or even other users. By completing some of the preferential user fields, such as nationality, age or preferred beverages, the users will also be able to regularly find beverages, offers, other users or reviews that are more suited to them. Instead of searching each time for a specific topic, the user will be presented with the option of checking a feed of recommendations in which popular or personally customized user or beverage related data can be seen. The application acts as a powerful beverage based search tool in which people interact with beverage data and each other by doing actions only related to beverages.

Web services, APIs and other technologies used as part of the application

In order to build our application we will use:

  • Stardog. This is our choice of a triplestore. We will store in Stardog our created knowledge base and use it in order to perform SPARQL queries, inferences, etc.
  • Google Container Registry & Google Cloud Run. We will use these services in order to deploy our knowledge base in Google Cloud. Starting with a local Stardog Docker image, we will store it in the cloud with Google Container Registry and we will further create and run the resulted container using Google Cloud Run.
  • Firebase. We will use Firebase in order to authenticate users and perform related tasks such as email verificationwhen a new user registers on our platform.

Application Use Case

The user is be provided with different tools for interacting with the application. The interaction implies either data searching, filtering, visualization, modification or creation.

User Account & Authentication

The be able to use the application to its full capacity the user is required to make an account, the process also includes the user as a semantic object into the database. After the creation of the account the user is be able to perform CRUD operations on his account data, operations that include even account termination.

Beverage and User Profile

The user is able to modify metadata associated with his account through his Beverage Profile Page while also be able to add Beverage Offers, Recipes, Scores or Reviews. All the actions that the user does will be used in the future to provide better Item Searches and Recommendations.

Item Search & Filter

The user can search through any type of item, including users and beverages, while also being able to filter the items by their corresponding data or metadata. Each Item will comply with the linked data principles and the application will also guide the user on how to access the data sources and related items.

Smart Item Recommendation

The users can view a feed of recommendations for each category of items. The feed, by default, will contain popular items, but can be switched to containing items that are similar to that type of items that are associated with the user.

External data sources used for building the knowledge base

In order to build the knowledge base for our project, we collected data from public API(s), open database(s) and other datasets. The used data sources are as follows:

  • Open Food Facts database
  • EDAMAM recipe API
  • Public (small) datasets (list of countries & nationalities)

Before creating the knowledge-base we also tested (and considered using) APIs such as:

  • Spoonacular API
  • CocktailDb API
  • FoodData Central API

Next, we will discuss only about the data sources actually used in our application.

Open Food Facts database

From here we extracted data regarding most common (not necessarily disjoint) categories of non-alcoholic beverage products. In case of beverages with a potential small alcohol content, we checked that the alcohol by volume value is below 1% (chosen convention). The table below summarizes the number of products extracted for each category.

In case of juices and nectars categories we further split them in subcategories (AppleJuice, PearNectar, etc.) so that we could correlate these drinks in the future with the associated names of fruits and vegetables. The data was downloaded in csv and excel formats.

EDAMAM recipe API

We used this API in order to extract data regarding non-alcoholic beverage recipes. We chose it because it offered a high amount of quality data (beverage recipes along with associated URI, image, ingredients, calories and detailed nutritional information). Recipes are also split by categories: Smoothie, Syrup, Tea, Coffee, Chocolate Drink, Juice, Milkshake. We automatically extracted the data using a script (because at one request we could only extract 20 recipes and we could only make 10 requests per minute). The total number of extracted recipes was 17714. The data was downloaded in excel format.

Public (small) datasets

We used datasets which offered us the list of world-wide countries and nationalities. We will use this data to provide metadata about users and beverages and to make correlations.

Considerations, expresiveness and integration aspects regarding the designed ontological model (updated)

We described categories of beverages through a hierarchy of classes (corresponding to categories from table 1). We opted for this type of modelling in order to ensure an easy extension of the ontology in the future, in case of adding other subcategories or categories of drinks. The root class of this taxonomy is class Beverage.

In case of beverages such as juices and nectar we decided to define subcategories by splitting these beverages based on the kind(s) of fruits and/or vegetables they contain (ex: AppleJuice, PearNectar, etc.). We correlated these subclasses with classes Fruit and Vegetable through object properties isMadeOfFruit and isMadeOfVegetable. Figure 9.B highlights the subclasses representing types of beverages:

Next, we will provide a general overview of classes and their associated properties (data properties and object properties) we used in our model:

  • class Person. Models the user information. This class is connected through object properties with classes Nationality, Country, HealthRestriction, meaning that a user with a certain nationality lives in a certain country and may have some health restrictions (such as lactose intolerance or diabetes). We also defined object properties to express the fact that a user may have one or more beverage preferences and/or one or more beverage category preferences (for example: a user prefers plant-based products). Person is also connected to classes Recipe, Offer and Review with object properties that express the fact that a user can add a review for a beverage, make a beverage offer or add scores for beverages and recipes. We will present below a mechanism for expressing the association between users, scores and beverages/recipes.
  • classes Country and Nationality. Useful for describing beverage provenance, the country where a user lives in or the user’s nationality. Therefore, both the residence country and nationality of the user are taken into account, in order to increase the expressiveness of our model.
  • class Health Restriction. Indicated health restrictions of users, such as intolerances to certain ingredients/substances (lactose, caffeine, gluten, fructose), certain diseases and health conditions that impact or are highly impacted by alimentation (eg.: diabetes, cancer).
  • class HealthLabel. Defines the health labels that are used to describe beverages and recipes. Health labels indicate the existence or non-existence of an important or potentially dangerous ingredient in the beverage or a certain property of the beverage, for example: milk-free, for-vegans, guarana-extract, etc.
  • class Ingredient. Associated with classes Beverage and Recipe, describes the ingredients that a beverage/recipe can have. Ingredients have a different naming convention from beverages and recipes, in order to avoid ambiguity (ex: water as beverage or as ingredient).
  • class Category. Defines the fine-grained categories that a beverage can belong to. We used the hierarchy of classes for defining main classes of beverages, but we also needed a mechanism to associate more expressive and/or detailed categories to beverages (such as squeezed-juices, instant-coffees, flavoured-syrups, etc.). These categories have been previously collected from the data extracted from Open Food Facts. It was almost impossible and impractical to model all of those categories as a set of classes, so we decided to create this new class to capture the small categories that a beverage can have. In this way we increased the expressiveness of our model.
  • class Beverage. We associated this class with classes Review and Recipe and Offer because a beverage can have associated one or more recipes/ reviews and offers posted by the users of the application. We also associated Beverage class with class Country through an object property that indicates the provenance(s) of the beverage. Beverage was also connected to class Category and to class Ingredient. Among beverage information is included the url from Open Food Facts where the beverage description can be found.
  • class Recipe. Recipes are associated with the beverage category (eg.: Smoothie, Milkshake, Tea, etc.) via a data property. This association was possible due to the fact that we applied filters for the category of beverage recipe before extracting recipes from EDAMAM. Recipe class also includes data properties such as recipe source (the website from where the recipe was posted) and recipe url to the original recipe from EDAMAM.
  • Common properties for beverages and recipes. Integration aspects. Both beverages and recipes have associated ingredients, nutritional information (22 indicators, such as calories, calciumContent, sugarContent, vitaminAContent, vitaminB6Content, etc.) and health labels. We created the set nutritional information and the set of health labels by correlating information regarding these that we found in Open Food Facts and Edamam. The association between the names of health labels (and the names of nutrition information indicators) was made by hand. The set of ingredients from both data sources was also merged. Thus, we created and used the same convention over both products and recipes.
  • class Review The Review class describes reviews about beverages posted by the users of the application. Data properties associated with this class include starRating and reviewDescription (the rating and actual review given by a user). Reviews are directly correlated to beverages via an object property.
  • class Offer The Offer class describes offers of beverages added by the users of the application. Data properties associated with this class include offerDescription, offerPrice, offerQuantity, priceCurrency and a property indicating the category of beverage for which the offer is posted (for example Lemon Juice/ Coffee).
  • class Score This class indicates the score that a user associates with different beverages and/or recipes. It helps in indicating the preferences of a user for certain beverages/recipes.
  • Integration between scores and beverages, recipes, reviews and offers The figure below depicts the association between scores and beverages, recipes, reviews and offers. Classes are denoted by circles, data properties by green rectangles, object properties by arrows, individuals by purple rectangles and naming convention of individuals by orange rectangles.

    Thus, offers are associated with beverage categories via isOfferFor data property. Relationship between offers and users who post the offer is made with the use of direct and inverse object properties: hasOfferOwner and ownsOffer. Each offer instance name includes the email of the user posting the offer and the beverage category for which the offer is posted (eg.: Tea, Coffee).

    Reviews are associated both to Person and Beverage class via object properties. The name of review instances includes the email of the user making the review and the code of the beverage for which the review is being made.

    Individuals from both Beverage and Recipe class can have associated scores. These scores are added by users in order to indicate their preference of certain beverages and/or recipes. The association is made via the Score class and object properties hasScore and its inverse, hasItemCode, that connect beverage and recipe instances with associated score instances. Classes Score and Person are also connected via a set of direct and inverse object properties. The name of score individuals includes the email of the user giving the score, the entity type for which the score is given (Recipe/Beverage) and the code of the item for which the score is attributed (a beverage/recipe code). Score class is also connected to Type property in order to search for and filter more efficiently scores for beverages from scores from recipes.

Recommendations Service

This service offers users three types of recommendations: beverage, recipe and (other) users recommendations. In case of all recommendations, the entity identifier (beverge/recipe code, user email) is returned along with a popularity indicator. The recommendations are of two types: simple & smart. We will detail below the smart type of recommendations.

Below is given a SPARQL query returning maximum 10 beverage recommendations for a specific user. In the inner "select" there are chosen beverages that either have an associated beverage category which is a category preferred by any user, or have an associated score, or have an associated review. The popularity of each beverage is calculated as the number of such beverages. In the end, there are selected beverages that have at least one category that is preffered by the given user. The results are sorted descending by popularity.

The SPARQL query below returns a list of maximum 10 recommended recipes along with their popularity score. In the inner "select" there are chosen only recipes that have at least one associated category. The popularity is calculated as the number of such recipes. Only recipes that have associated a category whose name resembles the name of a beverage class (which indicates a beverage type) are kept. The results are again sorted descending by popularity.

The last type of recommendation is user recommendations. The SPARQL query below retrieves a list of maximum 10 recommended users along with their popularity. Users are selected either if they have assigned scores for beverages and/or recipes, or if they have posted reviews and/or offers, or if they have a preferred category that is also a preferred category for at least another user. In the end, only users that have a common category with the user for which the recommendation is made are kept. The results are again sorted descending by popularity.

Data Modifications. Creating the knowledge base

We will further describe the data processing and aggregation steps that we had to go through in order to obtain the knowledge base.

Data enginnering

For the data to get to a state of being able to be put into the ontology many steps for sanitization, filtration and translation have been done. Each set of data, the one from the Open Food Facts and the one from Edamam, have been separately treated to respect the ontology structure as well as the data structures in which they will be stored and used.

Open Food Facts data

The raw Open Food Facts data was presented in a very bad initial state: lists as strings, text from various languages invalid numeric formats.. etc. The diagram bellow shows how the data has been sanitized, filtrated, transformed and translated to be able to fit the ontology predetermined criteria.

Words English Translation Algorithm

For translating the Open Food Facts texts the API from DeepL was used. The free capacity of characters translated provided by the API is 500k. Because of this limitation text was translated as words, not as full text. The algorithm, show in the diagram bellow, is build for character efficiency and data integrity while also providing a way to store the progress of translation in the middle of execution. The total amount of characters from the texts that needed to be translated was 2.4 millions, but with the help of the algorithm it was reduced to 470k, just within bounds for the DeepL API to be a viable solution.

Edamam data

The raw Edamam data was presented in a pretty good initial state, the only downside being the lists stored as strings. The diagram bellow shows how the data has been sanitized, filtrated and transformed to be able to fit the ontology predetermined criteria.

Aggregation and final outcome

The final step for creating our knowledge base was to aggregate the sanitized and translated datasets into our ontological model. In order to to this, we used the Cellfie Protégé plugin which converts spreadsheet data (excels) into OWL format. We created the transformation rules using MappingMaster language, a language which describes possible excel to OWL mappings. There was a total of 11 general rules (3 for recipe data, 6 for products data and 2 for nationalities and countries). The process of aggregation of all data was very slow (a few days) because of the large sizes of the datasets and the multitude of realized mappings.

How our solution conforms to Linked Data Principles

Below are some reasons regarding why we consider that our proposed solution follows the Linked Data Principles:

  • We use a knowledge model (an ontology) in order to model the data in our application (OWL is language is built on top of RDF, so the modelling respects the linked data principles).
  • When defining the ontological model, we tried as much as possible to reuse existing vocabularies and ontologies (details here). This increases reusability and increases the value of our model because of its interconnections with well-known vocabularies and ontologies.
  • We use URIs for identifying any resources from the knowledge base. Therefore, beverage products, recipes, offers, reviews and most of their properties (ex: images) are/will be all identified by URIs. We will also use URIs for identifying other resources (non-informational) that will be part of our application.
  • We use/will use HTTP schema when defining URIs for the resources in our application (for example, this is already true for all of the URIs in our knowledge base).
  • Users of the application will be provided various useful related metadata when accessing one of the resources in our application (For example, the user will have access to nutritional information, health labels when accessing a beverage). We will also include important related information (For example, for a given recipe, the user will have access to the website from where the recipe was collected). We already ensured we can do these things mentioned above because of the way we have designed our knowledge base.

Project Progress

Initial Goals

Introduction

The project started from the idea of having an app where users could socially interact with one another through the means of beverages and recipes from a large and wide variety. The plan was to have a very modular SOA architecture very adaptable and be future proof in terms of adaptability in every aspect: data, microservices, backend and frontend.

Requirements

Having a goal, we decided go through the requirement analysis stage with an end user mindset. This resulted in the following specifications:

The actors consist in a list of non managed entities:

  1. User Not Authenticated
  2. User Authenticated
  3. Email Services

And a series of managed entities:

  1. Stardog Database Manager
  2. Beverage Profile Microservice
  3. User Profile Microservice
  4. Search & Filter Microservice
  5. Recommendations Microservice
  6. Authentication
  7. Back End Gateway
  8. Front End Client

Each of the managed entities have their own requirements that were made in the end user behavior in mind, fact which allowed a lot of scalability and modifications at develop time.

The requirements for the Front End (also presented as Main Application) is to orchestrate the interaction between the user and the other services, and also to provide a user interface.

The ones for the Back End Gateway are to unite and manage all of the microservices and provide the upper layer (in this case, Front End) with an easier solution to access the data and semantics provided by the microservices

The Beverage Profile microservice had the initial requirements of:

  1. As an authenticated user, I want to be able to add/update/delete positive or negative feedback (score) regarding beverages
  2. As an authenticated user, I want to see the list of beverages for which I have submitted feedback
  3. As an authenticated user, I want to add/update/delete reviews regarding beverages
  4. As an authenticated user, I want to add/update/delete offers regarding beverages

The Search and Filter microservice had the initial requirements of:

  1. As an authenticated user, I want to search entities (beverages, users, reviews, offers etc..) while also having the possibility to filter the results at each step.
  2. As an authenticated user, I want to be able to save and access the filters for the current search as a personal search group.

The Recommendation microservice had the initial requirements of:

  1. As an authenticated user, I want to receive recommendations on beverages on my feed which are made based on similarities/correlations between my metadata and other users metadata
  2. As an authenticated user, I want to configure my recommendations with some filters

The User Profile microservice had the simple requirement to manage the metadata of the user non related to the beverages, such as age, name, surname etc..

Technical Requirements

By having a list of fixed requirements for the user behavior we were able to build the technical requirements and goals for each part of the project.
