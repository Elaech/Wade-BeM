Introduction

The user guide provides insights on how to use the software built within the project and a presentation video that describes all of the aspects present in the present as well as some of the technicalities and implementation.

The implemented architecture is meant to be used by both humans and robots by making use of the beverage manager front end client and of the backend gateway GraphQL API.

Target Audience

The main target audience of the application consists in the users that will manually interact with the front end of the application. The usage is defined by searching, filtering and viewing beverages, recipes, offers, reviews, recommendations or other users and creating offers, reviews, metadata, beverage and/or recipe scores.

Also part of the beach head, are the users AI's and robots that parse web content and try to obtain information from it or its corresponding interactions. The usage incorporates the one for the human user as well as the usage of the GraphQL API provided by the back end gateway.

BeM Client Usage

The front end is composed of several individual pages that each represent some form of entity behind the scenes.

Login & Register Page

The first page that will be accessed by the user will always be the login and register page where the user will be prompted to enter its credentials (Fig 4.A) or create an account if the email is not in the database (Fig 4.B).

Homepage

After successfully creating or accessing an account the user will be moved on the homepage. The upper part of this page contains an introduction and a small description of the application and its purpose.

The homepage can also be accessed, like any other page, from the navigation menu.

The lower part of the page contains the recommendations feed which has the functionality of providing beverage/user/recipe recommendations to the user. The behavior of this functionality can be controlled by changing the parameters that determine the recommendations type (beverage/user/recipe) the maximum number of displayed items and the "smart" recommendations check box that makes the recommendations returned to be tailored to the currently logged user by observing its user profile and the other activities the user has done in the application. Notice that the recommendations will not update until the press of the "Refresh" button.

User Profile Page

The user profile page is meant for the user to add more metadata about him, like the age, the country he lives in, the nationality, his health restrictions like "diabetic" or also found as "sugar-free" and his beverage category preferences.

All of the changes made will be sent to the database for updating on the press of the submit button and will impact from then on the recommendations and the search and filter page results.

Search and filter page

The search and filter page provides initially a table that contains the all of the entities from the selected item type, chosen from the navbar on navigation. Its functionalities come from searching by providing a keyword in the text input residing in the top right corner, sorting the items by any of the columns and as well as providing the option to filter the results by the value of each one of the column values. A quality of life feature is the reset on all the filters.

The table contains all the metadata associated with the beverages/reviews/offers/recipes, unlike the other pages, which makes the application respect the principles of Open Linked Data but in turn comes with much higher loading times due to the big amount of data.

Beverage Profile page

The bevegare profile page, which can also be accessed from the navigation bar, is split into three subpages, each managing its own type of entity: like with score, offer for a category of beverages and a review with a star rating regarding a beverage.

The user can view, update, create or delete his own set of the above mentioned entities by completing the corresponding form and clicking the submit button.

Each action done on this page, apart from viewing, alters the behavior of the recommendations and search and filter functionalities for all of the users.

Logout and Redirects

As previously mentioned the user actions are always passed through the authorization middleware provided by the Firebase API and if something were to happen that would invalidate a user action, the user will be automatically logged off.

Another problem that could occur is if the user tries to access routes that are not open; in this case the solution being a redirect to the login page, which will authorize the user and in turn transport him to the homepage if the authentication is still valid.

The user can manually log out of the application if he presses the logout button situated in the navbar.

By simply closing the web application tab the user will remain authenticated through the means of the firebase cookie, unless the cookies for the beverage manager domain are deleted.

GraphQL API Usage

Bellow, there can be seen two examples of how one might use the GrahQL API to acquire data directly from the back end gateway. There are many usages, such as data scrapping or integration with another app, that can be achieved very easily because of the GraphQL API provided by the gateway. A quality of life feature comes from the fact that the GraphQL technology used allows users to experiment in a graphical manner, making the whole thing self documented in a way which not many APIs can be.

All the other example of the GraphQL queries or mutations can be found within the final deliverables.

Use Cases Examples

Simple Human and Beverage Interaction

This is represented by the main scenario of the application and is put into practice by any user that wants to create an account and use our functionalities for searching beverages, recipes or other things that are related in order to gain more knowledge about the beverage world that could suit his tastes.

API Integration into another Project

Because the gateway is made the way it is, another developer might want to integrate our entire functionality into his own application. This is another powerful usecase that is put into the spotlight by the well implemented gateway and as well as the very big semantic database that we have built; all tied together by the Open Linked Data principles that were respected.

Data Scrapping or Collecting

Because the gateway is made the way it is, another person can, if he builds a scrapper, parse the entirety of our database and scrap whatever data is needed for their project. This can be done in real time, if the data amount needed is small or is needed at somewhat high time intervals but can also be compacted into a real data science dataset. The dataset usage could constitute to research on human and beverage behavior or just simple study the beverage and recipe types and their metadata.
