Abstract

This report describes preliminary considerations regarding our WADE project, Beverage Manager. We describe the general structure of our proposed solution, the technologies (Web services, APIs, triple store(s) etc.) that will be managed or used by our application. We also provide details about our ontological model and about the process of creating the knowledge base (collection, sanitization, translation and aggregation of data). We conclude by giving reasons why we think our solution conforms to Linked Data Principles.

Introduction

Our application will use a single knowledge base both for user and beverages (meta-)data storage. This knowledge base will interact with our Search & Filter, Beverage Profile, User Profile and Recommendations services. The process of data collection is described in section X, whereas the steps for creating the knowledge base are discussed in section Y. A description of the ontological model is given in section Z.

Web services, APIs and other technologies used as part of the application

In order to build our application we will use:

  • Stardog. This is our choice of a triplestore. We will store in Stardog our created knowledge base and use it in order to perform SPARQL queries, inferences, etc.
  • Google Container Registry & Google Cloud Run. We will use these services in order to deploy our knowledge base in Google Cloud. Starting with a local Stardog Docker image, we will store it in the cloud with Google Container Registry and we will further create and run the resulted container using Google Cloud Run.
  • Firebase. We will use Firebase in order to authenticate users and perform related tasks such as email verification when a new user registers on our platform.
